Agent · 语音

全双工流式 AI 语音交互管线
（AI Voice Agent）

独立开发 · 2026.05
电话级实时语音对话系统：STT → LLM → TTS 全链路流式传输，支持毫秒级实时打断（Barge-in），将语音助手从"回合制问答"推进到"边听边说"的拟人化双向交互。

一、项目背景与痛点

传统语音助手的交互是串行回合制：录完 → 识别 → 生成 → 合成 → 播放，每一环都在等上一环收尾。用户的真实体感是三个痛点：

  1. 迟滞——说完话要等数秒才有回应，对话节奏断裂；
  2. 抢话——AI 一旦开始播报就停不下来，用户无法插话；
  3. 机械——回复以整段为单位吐出，没有真人对话的流动感。

本项目基于开源语音框架进行底层重构，目标是把交互体验做到电话级：系统边听边说、用户随时打断、响应即时跟进。

三、核心技术实现

1. 多路由 TTS 架构与成本优化

  • 7+ 供应商统一路由：火山引擎、Edge TTS、ElevenLabs、OpenAI、MiniMax、Mistral、Gemini、NeuTTS，按场景与质量要求自动选路；
  • 核心交互场景深度集成火山引擎大模型语音服务；非核心场景与降级备用走 Edge/Azure 等免费方案，在保障体验的同时将 API 调用成本压到最低；
  • 流式 / 非流式混合调用策略：实时对话走流式合成，长文本播报走非流式批量合成，按需切换。

TTS 供应商降级链（路由逻辑）

用户配置 → Load TTS Config → Provider Chain
  ├─ 首选：用户指定 Provider
  │   ├─ 成功 → 直接返回
  │   └─ 失败 → 记录错误，进入 Fallback
  ├─ Fallback 1：次级 Provider
  │   ├─ 成功 → 返回（附带 degraded 标记）
  │   └─ 失败 → 继续降级
  ├─ Fallback 2：免费 Provider（Edge TTS）
  │   └─ 成功/失败 → 返回（零成本兜底）
  └─ 兜底：静默失败/错误提示

设计要点：每个 Provider 独立惰性加载，避免在无麦克风环境（SSH/Docker）崩溃。失败检测维度：HTTP 状态码、超时（10s）、配额耗尽、网络断开。降级标记自动上报监控日志。

2. 全链路流式传输

  • 打通 STT → LLM → TTS 的端到端流式管线：LLM 输出按句缓冲，每生成一个完整句即送入 TTS 合成并播放，无需等待全文生成；
  • 用户听到第一句话的时间从"全文生成完毕之后"提前到"第一句话生成完毕之时"，体感延迟大幅压缩。

句级缓冲调度

LLM 流式文本 → 缓冲区
  ├─ 累积字符 ≥ 20
  ├─ 检测句子边界（。！？\n 等分隔符）
  ├─ 剥离 Markdown 格式 + <think> 推理块
  └─ 送入 TTS 合成 → 实时播放
      └─ 同时继续累积后续文本，形成流水线

多级音频播放器

音频播放优先级：
  1. sounddevice.play()        — WAV 格式，低延迟
  2. afplay                    — macOS 原生
  3. ffplay -nodisp            — 跨平台（ffmpeg）
  4. aplay                     — Linux ALSA

3. VAD 两阶段静音检测

  • 阶段一（语音确认）：等待 RMS 幅度连续 ≥0.3s 高于阈值，过滤瞬时噪声；
  • 阶段二（结束检测）：语音确认后持续静默 ≥3.0s 触发停止；若 15s 内未检测到任何语音则超时自动结束。
  • 内置 dip 容忍机制——说话中途的短暂停顿（如换气、思索）不会被误判为结束。

dip 容忍算法详解

[待语音] → RMS > 阈值(200)
    ↓
[语音尝试中] → 连续 ≥0.3s 高于阈值 → [已确认说话]
    ↑  (若 RMS 下跌：启动 dip 计时器)
    ↓  (dip 持续 >0.3s → 重置回 [待语音])
    ↓
[已确认说话] → RMS < 阈值 → 启动静默计时器
    │              ↑ (说话恢复 ≥0.3s → 重置静默计时器)
    ↓
[静默 ≥3.0s] → 触发停止回调 → 转录流程
[超时保护] → 无语音 ≥15s → 自动结束
参数默认值说明
silence_threshold200 (RMS)低于此值视为静音
silence_duration3.0s静默多长时间触发停止
max_wait15.0s无语音超时上限
min_speech_duration0.3s确认说话的最短时长
max_dip_tolerance0.3sRMS 下跌宽容窗口

音频流管理：InputStream 常开不关，避免 macOS CoreAudio 下反复打开/关闭导致的挂死 Bug。

4. 实时打断（Barge-in）

用户在 AI 播报过程中随时插话，系统毫秒级执行三连动作：截断当前音频流 → 终止正在进行的 LLM 推理 → 重置对话上下文，彻底解决 AI 抢话问题。

# 全局播放引用（可中断）
_active_playback: subprocess.Popen | None = None

def stop_playback():
    # 1. 终止子进程播放器
    if _active_playback and _active_playback.poll() is None:
        _active_playback.terminate()
    # 2. 停止 sounddevice 播放
    sd.stop()
    # 3. LLM 推理通过全局中断标记停止

打断触发链：用户说话 → VAD 检测到语音 → 调用 stop_playback() 截断音频 → 设置 LLM 中断标记 → 清空 TTS 缓冲区 → 启动新录音循环。

回声防止：播放 TTS 时自动暂停麦克风监听。
