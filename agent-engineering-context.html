# 上下文工程与记忆管理 / Context Engineering & Memory


> **来源 / Source**：Anthropic Engineering — *Effective context engineering for AI agents*（2025-09-29）

> https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

> 作者 / Authors: Applied AI team — Prithvi Rajasekaran, Ethan Dixon, Carly Ryan, Jeremy Hadfield 等

> 中英对照 + 结构化要点，供知识库 / Agent 检索使用。



## 0. 核心定义 / Definition


> **EN**: Context engineering refers to the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference, including all the other information that may land there outside of the prompts. The guiding principle: find the **smallest possible set of high-signal tokens that maximize the likelihood of a desired outcome**.

>

> **中**：上下文工程 = 在 LLM 推理过程中，对最优 token（信息）集合进行**策划与维护**的一整套策略，涵盖 prompt 之外一切可能进入上下文的信息。指导原则：找到**能最大化目标达成概率的、最小的高信号 token 集合**。


> **与 prompt 工程的区别 / vs prompt engineering**：prompt 工程是写好指令（尤其 system prompt）的一次性任务；上下文工程是**迭代**的——每次决定"传什么给模型"时都要做一次策划（curation）。Context engineering 是 prompt engineering 的自然演进。



## 1. 为什么重要：上下文腐化 / Why it matters: context rot


> **EN**: LLMs, like humans, lose focus at a certain point. **Context rot**: as the number of tokens increases, the model's ability to accurately recall information decreases. Context must be treated as a **finite resource with diminishing marginal returns**. LLMs have an "attention budget" that every new token depletes.

>

> **中**：LLM 和人一样，到某个点会失焦。**上下文腐化（context rot）**：token 数越多，模型准确回忆信息的能力越下降。上下文必须被当作**边际收益递减的有限资源**。LLM 有"注意力预算"，每多一个 token 都在消耗它。


### 机制 / Mechanism

> **EN**: Transformer architecture enables every token to attend to every other token → **n² pairwise relationships** for n tokens. As context grows, the ability to capture these relationships gets stretched thin. Models also train on data where shorter sequences dominate, so they have fewer specialized parameters for context-wide dependencies. Result: a **performance gradient, not a hard cliff** — capable at long context but with reduced precision for retrieval and long-range reasoning.

>

> **中**：Transformer 架构让每个 token 关注其他所有 token → n 个 token 产生 **n² 对关系**。上下文增长时，捕捉这些关系的能力被摊薄。且模型训练数据里短序列占多数，处理全上下文依赖的专用参数更少。结果是**性能渐变而非断崖**——长上下文仍能用，但检索精度和长程推理会下降。


> ⚠️ 直接含义：**加大上下文窗口 ≠ 解决问题**。窗口再大也逃不过 context rot 和信息相关性问题。这是你（小A）做 Hermes 三层记忆持久化的底层依据——不能靠"把所有东西塞进去"，必须策划。



## 2. 有效上下文的解剖 / Anatomy of effective context


### System prompt：合适的"海拔" / the right altitude

> **EN**: System prompts should be at the "right altitude" — the Goldilocks zone between two failure modes: (1) hardcoding complex brittle if-else logic → fragility; (2) vague high-level guidance that gives no concrete signal. Optimal: specific enough to guide behavior, flexible enough to give strong heuristics.

>

> **中**：system prompt 要在"合适的海拔"——两个失败模式之间的黄金地带：①硬编码复杂脆弱的 if-else 逻辑 → 脆弱难维护；②含糊的高层指引 → 没有具体信号。最优：**足够具体能引导行为，又足够灵活能给强启发式**。


- 用 `<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` 等分区，配合 XML 标签或 Markdown 标题。

- 追求**完整描述预期行为的最小信息集**（minimal ≠ short，最小不等于短）。

- 先用最强模型 + 最小 prompt 测基线，再根据失败模式加指令和示例。


### Tools：定义 Agent 与世界的契约

> **EN**: Tools should be **self-contained, robust to error, extremely clear** in intended use. Most common failure: **bloated tool sets** with overlapping functionality / ambiguous decision points. If a human engineer can't say which tool to use, the agent can't either.

>

> **中**：工具应**自包含、容错、用途极其清晰**。最常见失败：**臃肿的工具集**——功能重叠、决策点含糊。如果人类工程师都说不清某情况该用哪个工具，别指望 Agent 能做得更好。


### Examples：few-shot

> **中**：保留一组**多样、典型（canonical）的示例**有效刻画预期行为，而不是把所有 edge case 塞进 prompt。对 LLM 而言，示例是"一图胜千言"的那张图。

> **EN**: Curate diverse, canonical examples — don't stuff a laundry list of edge cases.


> **总指引 / Meta**：所有组件（system prompt / tools / examples / message history）都要 **informative yet tight**（信息充分但紧凑）。



## 3. 运行时检索：Just-in-Time 策略 / Context retrieval


> **Agent 定义 / Definition adopted**：LLMs autonomously using tools in a loop（在循环中自主使用工具的 LLM）。


| 范式 | 要点（中） | Key point (EN) |

|---|---|---|

| 传统：预推理检索 | embedding-based，提前把相关数据处理好塞进上下文 | Embedding-based pre-inference-time retrieval |

| 新：Just-in-Time | Agent 只维护**轻量标识符**（文件路径、查询、链接），运行时用工具动态加载数据。类比人类认知：不背全部信息，而用文件系统/收件箱/书签按需取回 | Maintain lightweight identifiers (file paths, queries, links); load data dynamically at runtime. Mirrors human cognition |

| 混合 / Hybrid | 部分数据提前取（要速度）+ 部分自主探索（用 grep/glob 之类）。Claude Code 即用此模式：CLAUDE.md 提前进上下文，glob/grep 即时取文件 | Retrieve some data up front for speed; autonomous exploration at discretion. Claude Code: CLAUDE.md up front + glob/grep just-in-time |


### 元数据即信号 / Metadata as signal

> **EN**: To an agent in a file system, a file `test_utils.py` in `tests/` implies a different purpose than the same name in `src/core_logic/`. Folder hierarchies, naming conventions, timestamps all provide signals. This enables **progressive disclosure** — agents incrementally discover context through exploration.

>

> **中**：对在文件系统里操作的 Agent，`tests/` 下的 `test_utils.py` 与 `src/core_logic/` 下同名文件含义不同。文件夹层级、命名约定、时间戳都是信号。这实现**渐进式披露（progressive disclosure）**——Agent 通过探索逐层组装理解。


> **权衡 / Trade-off**：运行时探索比取预算数据慢，且需要精心的工程（给对工具和启发式），否则 Agent 会误用工具、追死胡同、漏掉关键信息。原则：**"do the simplest thing that works"（做能用的最简方案）**。



## 4. 长程任务三大技术 / Long-horizon techniques


> 长程任务 = token 数超过上下文窗口的连续动作序列（如大型代码库迁移、综合研究，跨数十分钟到数小时）。三种技术直接对抗上下文污染：


### ① Compaction（压缩重启）

> **EN**: Take a conversation nearing the limit, summarize it, reinitiate a new context window with the summary. In Claude Code: pass message history to the model to summarize; **preserve architectural decisions, unresolved bugs, implementation details; discard redundant tool outputs**. Continue with compressed context + 5 most recently accessed files.

>

> **中**：把接近上限的对话总结后，用摘要重启一个新上下文窗口。Claude Code 的做法：把消息历史交给模型总结，**保留架构决策、未解决的 bug、实现细节；丢弃冗余的工具输出**。然后用"压缩后上下文 + 最近访问的 5 个文件"继续。


> **艺术在于取舍**：压得太狠会丢掉当时不起眼、后来才显重要的上下文。工程建议：先**最大化召回**（确保抓全所有相关信息），再迭代**提升精度**（删冗余）。最安全的轻量压缩 = **清除工具调用结果**（tool result clearing）——工具结果用过一次后，深埋在历史里的原始结果没必要再看。


### ② Structured note-taking（结构化笔记 / agentic memory）

> **EN**: The agent regularly writes notes persisted to memory **outside** the context window, pulled back in later. Persistent memory with minimal overhead. E.g. Claude Code's to-do list, or a custom agent's NOTES.md.

>

> **中**：Agent 定期把笔记写到上下文窗口**之外**的记忆里，之后再拉回来。低开销的持久记忆。例：Claude Code 的 to-do 列表、自定义 Agent 的 NOTES.md。


> **案例 / Claude plays Pokémon**：Agent 跨数千游戏步维护精确计数（"过去 1234 步我在 1 号路练级，皮卡丘升了 8 级、目标 10 级"），自发画出已探索区域地图、记住成就、维护战斗策略笔记。**上下文重置后读自己的笔记继续**多小时训练序列。

> **memory tool**：Sonnet 4.5 起在 Claude Developer Platform 上线了基于文件系统的记忆工具（public beta），让 Agent 跨会话维护项目状态、积累知识库。


### ③ Sub-agent architectures（子 Agent 架构）

> **EN**: Specialized sub-agents handle focused tasks with **clean context windows**. Main agent coordinates with a high-level plan; subagents do deep work and return only a **condensed summary (often 1,000–2,000 tokens)** from tens of thousands of tokens of exploration. Clear separation of concerns.

>

> **中**：专职子 Agent 用**干净的上下文窗口**处理聚焦任务。主 Agent 用高层计划协调；子 Agent 可能用数万 token 深入探索，但只返回**浓缩摘要（通常 1000–2000 token）**。清晰的关注点分离。


### 三选一决策 / Which to choose

| 技术 | 适用 | When |

|---|---|---|

| Compaction | 需大量来回对话、要保持对话流 | tasks requiring extensive back-and-forth |

| Note-taking | 有明确里程碑的迭代式开发 | iterative development with clear milestones |

| Multi-agent | 复杂研究/分析、并行探索有回报 | complex research where parallel exploration pays |



## 5. 给小A的提炼（非原文，我的标注）


- 你的"三层记忆持久化架构"（重启禁令规则 + 启动配置 + HERMES.md→prefill.json 转换脚本）= 本文 **structured note-taking + compaction** 的组合落地。本文给你补的依据：①context rot 证明"加大窗口"不是解法，外化记忆才是；②压缩要"先召回后精度"的调参顺序；③tool result clearing 是最安全的轻量压缩，你 Hermes 可优先上这个。

- 你 Hermes 的"工具优先级 / 会话复用"规则，对应本文第 2.2 节（工具是 Agent 与世界的契约、忌臃肿工具集）和第 3 节（just-in-time + 会话内元数据信号）。

- "合适的海拔"这个概念可直接用于你给 DeepSeek 写 prompt：你之前说 DeepSeek 过度顺从、prompt 改不动——本文提示，问题可能不在"压不住"，而在 prompt 海拔过低（含糊）或过高（脆弱硬编码）。先校准海拔再判断要不要上架构强制。
